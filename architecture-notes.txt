ROOM BOOKING SYSTEM - ARCHITECTURE NOTES
=========================================

1. HOW OVERLAPPING BOOKINGS ARE PREVENTED
------------------------------------------
The system prevents double bookings using SQL-based date overlap detection:

Logic:
- When a user attempts to book a room, the backend checks for existing bookings
- Overlap condition: (start_date <= new_end_date) AND (end_date >= new_start_date)
- This catches all possible overlap scenarios:
  * New booking completely contains existing booking
  * Existing booking completely contains new booking
  * Partial overlap at start or end
  * Exact date match

Implementation:
- Database query runs before creating booking
- Uses database transactions to ensure atomicity
- If overlap found, returns 400 error with message
- Only creates booking if no conflicts exist

Benefits:
- Database-level validation ensures consistency
- Race conditions handled by transaction isolation
- Index on (room_id, start_date, end_date) for performance


2. HOW FRONTEND COMMUNICATES WITH BACKEND
------------------------------------------
Communication Pattern:
- RESTful API architecture
- JSON data format for all requests/responses
- Axios library for HTTP requests
- JWT token-based authentication

Flow:
1. User logs in → Backend returns JWT token
2. Frontend stores token in localStorage
3. All subsequent requests include token in Authorization header
4. Backend middleware validates token before processing requests
5. Protected routes redirect to login if token missing/invalid

API Service Layer:
- Centralized API calls in services/api.js
- Automatic token attachment via Axios interceptors
- Error handling and response formatting
- Base URL configuration for environment flexibility


3. WHY THIS FOLDER STRUCTURE WAS CHOSEN
----------------------------------------
Backend Structure:
/config       - Database and environment configuration (separation of concerns)
/controllers  - Request handling and response logic (thin layer)
/services     - Business logic and database operations (reusable)
/middleware   - Authentication and validation (cross-cutting concerns)
/routes       - API endpoint definitions (clear API surface)

Benefits:
- Separation of concerns (MVC-like pattern)
- Easy to test individual components
- Services can be reused across controllers
- Middleware can be applied to multiple routes
- Clear responsibility for each layer

Frontend Structure:
/components   - Reusable UI components (Navbar, RoomCard, etc.)
/pages        - Full page components (Login, Dashboard, etc.)
/services     - API communication layer

Benefits:
- Component reusability
- Clear separation between pages and components
- Centralized API logic
- Easy to add new features
